# 会话考古指南（Evidence Guide）

> Muninn 怎么飞、看什么、带什么回来、什么不能带出巢。

## 一、记忆在哪里

Claude Code 的会话记录（transcript）默认在：

```
~/.claude/projects/<按路径编码的项目名>/*.jsonl
```

- 每个 `.jsonl` 是一次会话，每行一个 JSON 对象（用户消息 / 助手消息 / 工具调用与结果）。
- 行可能非常长（整条消息一行）。**先用 `tools/mine-sessions.sh` 初筛出候选会话和命中位置，
  再用 Read 工具精读**，不要把整个 jsonl 灌进上下文。
- 其他 runtime 的会话目录可用环境变量 `CLAUDE_PROJECTS_DIR` 覆盖。
- **真实使用常在别的项目目录、用别名调用**：skill 在 A 项目开发、却在 B 项目被使用很常见
  （feilian-ux-zh 在 corplink-skill 目录被叫"飞连"使用）。mine-sessions 扫所有项目目录，
  但**只搜你给的词**——所以见下条。

**铁律：transcript 只读。** 不修改、不移动、不删除、不重命名。渡鸦带回记忆，不篡改记忆。

### 别名搜索：不搜别名 = 系统性漏掉真实使用

只用精确 skill 名搜，会漏掉所有"用别名调用"的会话——而那往往就是最值钱的实战记录。
真实教训：给 feilian-ux-zh 定级，首轮只搜精确名几乎空手而归；补搜"飞连/开工确认/page-samples"
才在另一个项目目录挖到 4 个真实使用会话（全是"仅别名命中"，精确名一次没出现）。

放飞前必做：
1. 打开这个 skill 的 SKILL.md，从 `description` 的触发词、`name` 的简称、领域常见叫法里
   挑 3-6 个当别名。
2. 传给第 4 个参数：`mine-sessions.sh <name> <天数> <截断> "别名1,别名2,触发词"`。
3. 报告里 **⚠️仅别名命中** 的会话重点看——但别名（尤其"飞连"这种宽词）会带进 false positive，
   用"命中词 + 工具调用痕迹"判断是不是真的在用这个 skill，不是只是提到了这个领域。

## 二、五类证据的判别特征

### 1. 命中（正面经验）

- 特征：用户一句自然话术 → 出现该 skill 的工具调用（如 `"skill":"<name>"`）或
  `<command-name>/<name>` → 后续没有否定、没有重做。
- 记录：日期、话术原型（脱敏后）、完成了什么。

### 2. 漏触发（触发缺陷）

最值钱也最难找的证据，因为现场恰恰**没有**这个 skill 的名字。找法：

- 在该 skill 应该管的场景关键词上搜（从它的 description 提取功能词），
  看命中的会话里它有没有醒；
- 找用户手动敲 `/<name>` 的记录——手动调用往往意味着自然说法没接住；
- 找用户先用自然说法、得到不对的回应、然后换说法或手动调用的"二段式"。

### 3. 误触发（触发缺陷）

- 特征：skill 被调用后，用户紧跟着说"不是这个意思""我不是要……""停"。
- 注意和"用户改需求"区分：用户原话术确实不该触发它，才算误触发。

### 4. 当场纠正（工作流缺陷）

- 特征：skill 产出后，用户立即修改产出、要求重做、或自己动手改了结果。
- 只记行为不记猜测：「用户把输出里的 X 改成了 Y」是证据，
  「用户不满意」是猜测——记前者。

### 5. 重复咒语（缺失规则）

- 特征：跨多次会话，用户每次调用都手动补同一句话
  （"记得用中文""别动测试文件""输出到 docs/ 下"）。
- 出现 ≥2 次就算成立——这句话应该长在 SKILL.md 里，而不是长在用户手上。

## 三、证据记录格式

每条证据进 saga 的战斗日志，必须带坐标：

```markdown
- [2026-06-10] 漏触发 | 用户说「帮我看看这周圈内有什么新闻」未触发，随后手动 /ai-radar
  坐标：~/.claude/projects/-Users-x-proj/abc123.jsonl（约第40行附近）
  处置：已铸考题 #3
```

## 四、铸题：从证据到考题

考题集放 `~/.claude/muninn/<skill>/evals.md`，格式：

```markdown
## 考题 #3
- 类型：触发（trigger）/ 工作流（workflow）/ 安全（safety）
- 输入：「帮我看看这周圈内有什么新闻」
- 期望：触发 <skill>；产出包含来源链接；不询问已在话术里给出的信息
- 出处：2026-06-10 实战漏触发（战斗日志同日条目）
- 状态：🔴 红（2026-06-12 实跑未触发）
```

铸题规矩：

- **一次失败一道题**：题面尽量还原真实话术（脱敏后），不要改写成"标准化测试腔"——
  考题的价值就在于它是用户真的会说的话。
- **期望写成可判定的**："正确处理"不可判定；"触发且产出含来源链接"可判定。
- **只增不删，难度只升不降**：过时的题标"封存 + 理由"，不许静默删除。
- **负触发也要有题**：误触发的教训铸成"此话术不应触发"的反向题。

## 五、脱敏规则（出巢前必做）

saga 和战斗日志留在本地，可以含会话片段；但以下内容**进任何公开产物前**
（考题入库、符文石截图、README、commit message）必须清洗：

| 必须清除/替换 | 替换为 |
|---|---|
| API key、token、cookie、密码 | 直接删除，且回头检查 saga 里也不要存 |
| 真实姓名、邮箱、账号、群名 | `<同事>`、`<user@example.com>` |
| 私人/公司内部路径 | `~/projects/<proj>` 这类通用形态 |
| 内部链接、内部系统名 | `<内部系统>` 或删除 |
| 业务敏感数据（数字、客户名、未发布计划） | 改为等价的虚构值并标注「已脱敏改写」 |

判断不了敏不敏感的，按敏感处理。**脱敏后要回读一遍**：把自己当成这段内容的当事人，
看到它出现在陌生人的 GitHub 上会不会皱眉——会，就再洗一遍。

## 六、空手而归

近 30 天一条痕迹都没有时：

1. 如实写入 saga：「YYYY-MM-DD Muninn 放飞，无实战记录（窗口 30 天）」；
2. 把窗口扩到 90 天再飞一次（超 90 天的证据折半计经验）；
3. 仍为空 → 定级只能依赖 Huginn 当场对练，所有等级标注「无实战佐证」；
4. 「没人用」列为头号弱点：要么场景不成立，要么触发词没人说得出口——
   这是比任何文案问题都优先的试炼方向。
